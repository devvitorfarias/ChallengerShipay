# 📋 Questão 1 — Validação de Cadastro CNPJ + CEP

## O que faz

Serviço síncrono que recebe um **CNPJ** e um **CEP**, consulta duas APIs externas em **paralelo** via Virtual Threads (Java 21), e verifica se o endereço da empresa bate com o endereço do CEP informado.

```
POST /v1/validation/cnpj-cep
```

| Resultado | HTTP | Body |
|---|---|---|
| Endereços coincidem | `200 OK` | `{ "valid": true }` |
| Endereços divergem | `404 Not Found` | `{ "valid": false }` |
| CNPJ/CEP inválido | `400 Bad Request` | mensagem de erro |
| API externa indisponível | `503 Service Unavailable` | mensagem de erro |

> **A resposta é sempre síncrona.** As chamadas ao CNPJ e ao CEP são disparadas ao mesmo
> tempo via `CompletableFuture`. O `.join()` bloqueia até ambas retornarem com virtual
> threads esse custo é baixo, mas a latência total ainda depende da chamada mais lenta.
> Para um modelo totalmente não-bloqueante, `Mono.zip()` com WebFlux seria a alternativa.

## Virtual Threads (Java 21)

Sem thread pool manual, sem WebFlux. O executor de virtual threads do Java 21 gerencia tudo:

// CepValidationService.java
private final ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();

public boolean validate(CepValidationRequest request) {
    var cnpjFuture = CompletableFuture.supplyAsync(() -> fetchCnpj(request.cnpj()), executor);
    var cepFuture  = CompletableFuture.supplyAsync(() -> fetchCepWithFallback(request.cep()), executor);

    return addressesMatch(cnpjFuture.join(), cepFuture.join());
}

## Virtual Threads (Java 21)

Sem thread pool manual, sem WebFlux. O executor de virtual threads do Java 21 gerencia tudo:
```java
// CepValidationService.java
private final ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor();

public boolean validate(CepValidationRequest request) {
    var cnpjFuture = CompletableFuture.supplyAsync(() -> fetchCnpj(request.cnpj()), executor);
    var cepFuture  = CompletableFuture.supplyAsync(() -> fetchCepWithFallback(request.cep()), executor);

    return addressesMatch(cnpjFuture.join(), cepFuture.join());
}
```

### Por que Virtual Threads aqui?

As chamadas ao CNPJ e ao CEP são operações de I/ a maior parte do tempo é espera de resposta
da API externa, não processamento de CPU. Virtual threads foram projetadas exatamente para esse
cenário: enquanto a thread aguarda o retorno HTTP, a JVM a desmonta da platform thread e a remonta
quando a resposta chegar, sem custo de bloqueio real.

### `newVirtualThreadPerTaskExecutor()` vs thread pool fixo

| | `newVirtualThreadPerTaskExecutor()` | `Executors.newFixedThreadPool(n)` |
|---|---|---|
| Custo por thread | ~1KB (heap) | ~1MB (stack nativa) |
| Limite prático | Milhões | Dezenas/centenas |
| Quando bloqueia I/O | JVM libera a platform thread | Platform thread fica ocupada |
| Tuning necessário | Nenhum | Escolher `n` correto |

Com virtual threads não há motivo para reusar threads , criar uma por tarefa é o padrão recomendado.

### O que o `.join()` faz (e o que não faz)
```java
// Dispara as duas chamadas ao mesmo tempo
var cnpjFuture = CompletableFuture.supplyAsync(..., executor);  // → virtual thread A
var cepFuture  = CompletableFuture.supplyAsync(..., executor);  // → virtual thread B

// Aguarda ambas terminarem antes de comparar
return addressesMatch(cnpjFuture.join(), cepFuture.join());
//                                 ↑
//                    bloqueia a thread chamadora até A terminar,
//                    depois até B terminar — mas B provavelmente
//                    já terminou enquanto esperávamos A
```

### Trade-off assumido

> Modelo síncrono com Spring MVC foi mantido intencionalmente. Introduzir WebFlux apenas para
> paralelizar duas chamadas trocaria o framework inteiro por um ganho pontual. Virtual threads
> entregam o benefício de paralelismo com I/O sem mudar o modelo de programação.
> I/O (Input/Output) é qualquer operação onde o programa envia ou recebe dados de fora da CPU e precisa esperar pela resposta.

## Resiliência

### Retry + Circuit Breaker via anotação
```java
@CircuitBreaker(name = "cnpj-service")
@Retry(name = "cnpj-service")
public CnpjResponse fetchCnpj(String cnpj) {
    return cnpjClient.findByCnpj(cnpj);
}

@CircuitBreaker(name = "cep-primary")
@Retry(name = "cep-primary")
public CepResponse fetchCepPrimary(String cep) {
    return cepPrimaryClient.findByCep(cep);
}
```

> 💡 **Por que anotações?**
> O Spring AOP (Aspect-Oriented Programming) intercepta a chamada e aplica Retry + Circuit Breaker sem que o método
> conheça a infraestrutura de resiliência.
>
> **A ordem é intencional:** `@CircuitBreaker` externo garante que o CB contabilize
> o resultado final dos retries , não cada tentativa individual. Se invertido,
> o circuito abriria muito mais rápido do que o esperado.
>
### Fallback automático entre provedores de CEP

Após Retry + Circuit Breaker do primário esgotarem, o fallback é acionado de forma transparente:
```java
public CepResponse fetchCepWithFallback(String cep) {
    try {
        return fetchCepPrimary(cep);                        // ViaCEP (primário)
    } catch (Exception e) {
        log.warn("Primary CEP failed, switching to fallback");
        return fetchCepFallback(cep);                       // BrasilAPI (fallback)
    }
}
```

### Por que `try/catch` explícito em vez de `fallbackMethod` do Resilience4j?

O Resilience4j oferece uma alternativa nativa para fallback:
```java
// Alternativa com fallbackMethod nativo:
@CircuitBreaker(name = "cep-primary", fallbackMethod = "fetchCepFallback")
public CepResponse fetchCepPrimary(String cep) { ... }
```

A abordagem com `try/catch` foi escolhida porque:

- **Legibilidade**: o fluxo primário → fallback está explícito em um único lugar,
  sem precisar navegar entre métodos para entender o comportamento
- **Controle**: é possível inspecionar a exceção antes de acionar o fallback,
  logar contexto específico ou tomar decisões diferentes por tipo de erro
- **Testes**: mais fácil de mockar e verificar em testes unitários
  sem depender do proxy do Spring AOP

> ⚠️ **Trade-off:** o `try/catch` captura `Exception` de forma ampla. Em produção,
> seria mais seguro restringir para `FeignException` e `IOException`, os mesmos
> tipos declarados no `retryExceptions` do `application.yml` ,  evitando engolir
> erros inesperados como `NullPointerException` que deveriam estourar.

### Configuração (`application.yml`)

```yaml
resilience4j:
  retry:
    instances:
      cep-primary:
        maxAttempts: 3
        waitDuration: 200ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2   # 200ms → 400ms → 800ms
        retryExceptions:
          - feign.FeignException
          - java.io.IOException
      cnpj-service:
        maxAttempts: 2
        waitDuration: 300ms

  circuitbreaker:
    instances:
      cep-primary:
        slidingWindowSize: 10
        failureRateThreshold: 50          # abre com 50% de falhas
        waitDurationInOpenState: 5s       # tenta reabrir após 5s
        permittedNumberOfCallsInHalfOpenState: 3
        minimumNumberOfCalls: 5
        registerHealthIndicator: true
      cnpj-service:
        slidingWindowSize: 10
        failureRateThreshold: 50
        minimumNumberOfCalls: 5
        registerHealthIndicator: true
```

### Fluxo de resiliência do CEP

```
Req → ViaCEP (tentativa 1)
         ├─ sucesso → retorna resposta ✅
         └─ falha → aguarda 200ms
              └─ ViaCEP (tentativa 2)
                   ├─ sucesso → retorna resposta ✅
                   └─ falha → aguarda 400ms (backoff exponencial)
                        └─ ViaCEP (tentativa 3)
                             ├─ sucesso → retorna resposta ✅
                             └─ falha → CB registra falha
                                  └─ catch → BrasilAPI (fallback)
                                       ├─ sucesso → retorna resposta ✅
                                       └─ falha → HTTP 503 ❌
```

> ⚠️ **Trade-off de latência:** no pior caso (tudo falha), o tempo acumulado antes do
> fallback é `200ms + 400ms + tempo das 3 chamadas`. Por isso o backoff exponencial
> é preferível ao intervalo fixo , respeita APIs sobrecarregadas e reduz a latência
> do pior caso.

## Normalização de Endereços

Usa `java.text.Normalizer` (JDK nativo, sem dependência extra) para remover acentos de forma robusta:

```java
// ❌ Antes — manual e incompleto:
.replaceAll("[àáâãä]", "a")
.replaceAll("[èéêë]",  "e")
// ...

// ✅ Depois — NFD decomposition, cobre todos os casos:
private String normalize(String value) {
    if (value == null) return "";
    return Normalizer.normalize(value.trim(), Normalizer.Form.NFD)
        .replaceAll("\\p{InCombiningDiacriticalMarks}+", "")  // remove todos os diacríticos
        .toLowerCase(Locale.ROOT)
        .replaceAll("[^a-z0-9 ]", " ")
        .replaceAll("\\s+", " ")
        .trim();
}
```

> `NFD` decompõe `ã` → `a` + diacrítico `~`. O regex `\p{InCombiningDiacriticalMarks}` remove
> todos de uma vez, cobrindo qualquer idioma sem listar cada vogal.

### Critério de comparação

| Campo | Critério | Exemplo |
|---|---|---|
| UF | Igualdade exata | `"sp"` == `"sp"` ✅ |
| Cidade | Igualdade exata | `"sao paulo"` == `"sao paulo"` ✅ |
| Logradouro | `contains()` bidirecional | `"paulista"` ∈ `"avenida paulista"` ✅ |

UF e cidade usam igualdade exata pois APIs diferentes retornam esses campos de forma
consistente após a normalização. Logradouro é mais permissivo porque o mesmo endereço
pode vir escrito de formas distintas dependendo da fonte:
```
ViaCEP  → "Avenida Paulista"
CNPJ.ws → "Paulista"
```

O `contains()` bidirecional resolve isso — independente de qual API retornou o nome
completo, o match funciona:
```java
boolean streetMatch = cnpjStreet.contains(cepStreet)
                   || cepStreet.contains(cnpjStreet);
```

> ⚠️ **Limitação conhecida:** `contains()` é permissivo logradouros muito curtos
> como `"rua"` podem gerar falso positivo. Uma evolução seria usar **Apache Commons Text**,
> adicionando a dependência no `pom.xml`:
>
> ```xml
> <dependency>
>     <groupId>org.apache.commons</groupId>
>     <artifactId>commons-text</artifactId>
>     <version>1.12.0</version>
> </dependency>
> ```
>
>
> ```java
> import org.apache.commons.text.similarity.JaroWinklerSimilarity;
> 
> var similarity = new JaroWinklerSimilarity();
> 
> double score = similarity.apply("avenida paulista", "paulista"); 

>
> > O threshold  seria ajustável conforme os casos reais de divergência entre APIs.
> Porém adicionaria uma dependência e lógica de calibração fora do escopo deste serviço.

## Feign Clients

```java
// Primário — ViaCEP
@FeignClient(name = "cep-primary-client", url = "${external.api.cep.primary.url}")
public interface CepPrimaryClient {
    @GetMapping("/{cep}/json")
    CepResponse findByCep(@PathVariable String cep);
}

// Fallback — BrasilAPI
@FeignClient(name = "cep-fallback-client", url = "${external.api.cep.fallback.url}")
public interface CepFallbackClient {
    @GetMapping("/{cep}")
    CepResponse findByCep(@PathVariable String cep);
}

// CNPJ
@FeignClient(name = "cnpj-client", url = "${external.api.cnpj.url}")
public interface CnpjClient {
    @GetMapping("/{cnpj}")
    CnpjResponse findByCnpj(@PathVariable String cnpj);
}

```
> ⚠️ **Limitação conhecida:** `Para o contexto do projeto — Spring Boot + Spring Cloud + Resilience4j — Feign ainda é a escolha mais pragmática. O HttpClient nativo seria superior em um projeto sem Spring Cloud, mas aqui exigiria reimplementar tudo que o Feign entrega automaticamente: serialização, retry, circuit breaker, configuração via YAML.`:
```yaml
# application.yml
external:
  api:
    cnpj:
      url: https://publica.cnpj.ws/cnpj
    cep:
      primary:
        url: https://viacep.com.br/ws
      fallback:
        url: https://brasilapi.com.br/api/cep/v2
```

O Circuit Breaker expõe seu estado no health check:

```bash
GET /actuator/health
```

```json
{
  "components": {
    "circuitBreakers": {
      "details": {
        "cep-primary":  { "state": "CLOSED" },
        "cep-fallback": { "state": "CLOSED" },
        "cnpj-service": { "state": "CLOSED" }

    }
  }
}

CLOSED  → circuito fechado  → requisições passam normalmente   ✅
OPEN    → circuito aberto   → requisições são bloqueadas        ❌
```
