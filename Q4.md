# 🔍 Questão 4 — Anti-Patterns identificados
---

##  `Tools.java` — 12 anti-patterns

### AP-01 · Credenciais hardcoded 

```java
// ❌ Token e credenciais expostos no código-fonte
private String headers = "Authorization: Bearer 123";
// e nos métodos JDBC:
Connection conn = DriverManager.getConnection(
    "jdbc:postgresql://127.0.0.1:5432/db", "user", "pass");
```

Qualquer pessoa com acesso ao repositório tem acesso às credenciais de produção.
O correto é externalizar via `application.yml` + variáveis de ambiente:

```java
// ✅
@Value("${chat.service.token}")
private String chatToken;
```

---

### AP-02 · SQL Injection em 3 métodos 

```java
// ❌ Concatenação direta de parâmetro na query — SQL Injection garantido
ResultSet rs = stmt.executeQuery(
    "SELECT access_key FROM secrets WHERE id = " + id);

ResultSet rs = stmt.executeQuery(
    "SELECT role_id FROM " + table + " WHERE entity_type = '" + entityType + "'");

ResultSet rsUser = stmt.executeQuery(
    "SELECT entity_type FROM users WHERE id = " + userId);
```

Um atacante pode passar `id = 1 OR 1=1` e obter todos os registros da tabela `secrets`.
Eu usaria `PreparedStatement`:

```java
// ✅
PreparedStatement stmt = conn.prepareStatement(
    "SELECT access_key FROM secrets WHERE id = ?");
stmt.setInt(1, id);
```

---

### AP-03 · Resource leak — Connection/Statement/ResultSet nunca fechados 

```java
// ❌ Conexão aberta, nunca fechada — esgota o pool em produção
Connection conn = DriverManager.getConnection(...);
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(...);
// sem conn.close(), stmt.close(), rs.close()
```

Em produção, cada chamada abre uma nova conexão com o banco que nunca é devolvida.
O pool esgota e o serviço para de responder. O correto é usar `try-with-resources`:

```java
// ✅
try (Connection conn = dataSource.getConnection();
     PreparedStatement stmt = conn.prepareStatement(sql)) {
    // uso
} // fecha automaticamente
```

---

### AP-04 · JDBC manual ignorando o DataSource do Spring 

```java
// ❌ Bypassa o pool de conexões gerenciado pelo Spring
Connection conn = DriverManager.getConnection(
    "jdbc:postgresql://127.0.0.1:5432/db", "user", "pass");
```

O Spring gerencia um pool de conexões (HikariCP) configurado e monitorado.
Usar `DriverManager` cria conexões avulsas, sem pool, sem métricas, sem controle de transação:

```java
// ✅ Injeta o DataSource gerenciado pelo Spring
@Autowired
private DataSource dataSource;
```

---

### AP-05 · JSON construído por concatenação de String 

```java
// ❌ Vulnerável a JSON injection se msgContent contiver aspas ou caracteres especiais
String body = "{\"message\": \"" + msgContent + "\"}";
```

Se `msgContent = "hello\" , \"injected\": \"value"`, o JSON fica malformado ou injetado.
O correto é usar Jackson:

```java
// ✅
String body = objectMapper.writeValueAsString(Map.of("message", msgContent));
```

---

### AP-06 · `HttpClient` criado a cada chamada 

```java
// ❌ Nova instância a cada invocação — overhead desnecessário
public String sendInstantMessage(String msgContent) throws Exception {
    HttpClient client = HttpClient.newHttpClient();
    ...
}
```

`HttpClient` é thread-safe e deve ser reutilizado como `@Bean` ou atributo da classe:

```java
// ✅
private final HttpClient httpClient = HttpClient.newHttpClient();
```

---

### AP-07 · `Tools` não é gerenciado pelo Spring 

```java
// ❌ Classe utilitária instanciada manualmente — fora do contexto Spring
public class Tools { ... }
// usada como: private final Tools tools; (injetada via construtor do Manager)
// mas a própria classe não tem @Component/@Service
```

Sem `@Component`, o Spring não gerencia o ciclo de vida, não injeta dependências
e não aplica AOP (logs, transações, métricas):

```java
// ✅
@Component
public class Tools { ... }
```

---

### AP-08 · `getSecretsById` retorna dado sensível como `String` 

```java
// ❌ Segredo trafega em memória como String plain text
// e pode aparecer em logs, stack traces e heap dumps
public String getSecretsById(int id) throws Exception {
    ...
    return rs.getString("access_key");
}
```

Segredos deveriam ser encapsulados e nunca expostos como `String` diretamente.

---

### AP-09 · `validateCnpj` valida apenas tamanho 

```java
// ❌ Qualquer sequência de 14 números passa como CNPJ válido
public boolean validateCnpj(String cnpj) {
    String cleaned = cnpj.replaceAll("[^0-9]", "");
    return cleaned.length() == 14;
}
```

Não valida os dígitos verificadores — `00000000000000` passa como válido.

---

### AP-10 · `getRoleByEntityType` tem lógica duplicada com o nome da tabela

```java
// ❌ Nome da tabela montado em if/else com string literal
String table = "users";
if (entityType.equalsIgnoreCase("admins")) {
    table = "admins";
}
ResultSet rs = stmt.executeQuery("SELECT role_id FROM " + table + ...);
```

Além do SQL Injection, a lógica de mapear `entityType` → tabela deveria estar
em um `enum` ou mapa estático, não em if/else com strings soltas.

---

### AP-11 · `getClaimsByUserId` faz N+1 queries 

```java
// ❌ Busca entity_type em uma query, depois role_id em outra, depois claims em outra
ResultSet rsUser = stmt.executeQuery("SELECT entity_type FROM users WHERE id = " + userId);
int roleId = getRoleByEntityType(type); // abre nova conexão
ResultSet rsClaims = stmt.executeQuery("SELECT meta_data FROM claims WHERE role_id = " + roleId);
```

Três queries separadas onde um único `JOIN` resolveria com menos round-trips ao banco.

---

### AP-12 · `Tools` acumula responsabilidades sem relação 

Uma única classe faz: envio de mensagens HTTP, queries JDBC de segredos,
queries JDBC de roles, queries JDBC de claims e validação de CNPJ.
Violação direta do **Single Responsibility Principle** — deveria ser dividida em
`ChatClient`, `SecretsRepository`, `RoleRepository` e `CnpjValidator`.

## `GlobalExceptionHandler.java` — 4 anti-patterns

### AP-13 · Stack trace exposto no response HTTP 

```java
// ❌ Qualquer usuário da API recebe o stack trace completo
ex.printStackTrace(pw);
String errorMessage = "Exception: " + ex.getMessage() + " - StackTrace: " + stackTrace;
return new ResponseEntity<>(errorMessage, HttpStatus.INTERNAL_SERVER_ERROR);
```

Stack traces revelam: estrutura interna do código, versões de bibliotecas,
nomes de classes e métodos — informações valiosas para um atacante.
O correto é logar internamente e retornar mensagem genérica ao cliente:

```java
// ✅
log.error("Unhandled exception", ex); // log interno
return ResponseEntity.internalServerError()
    .body(new ErrorResponse("internal_error", "An unexpected error occurred."));
```

---

### AP-14 · Captura `Exception` genérica demais 

```java
// ❌ Captura qualquer Throwable, inclusive erros de JVM
@ExceptionHandler(Exception.class)
public ResponseEntity<String> handleAllExceptions(Exception ex) { ... }
```

`NullPointerException`, `OutOfMemoryError`, `StackOverflowError` — todos caem aqui
e retornam 500 ao cliente sem distinção. O correto é ter handlers específicos:

```java
// ✅
@ExceptionHandler(DataNotFoundException.class)
public ResponseEntity<ErrorResponse> handleNotFound(DataNotFoundException ex) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND)
        .body(new ErrorResponse("not_found", ex.getMessage()));
}
```

---

### AP-15 · Sempre retorna HTTP 500

```java
// ❌ Um recurso não encontrado retorna 500, não 404
return new ResponseEntity<>(errorMessage, HttpStatus.INTERNAL_SERVER_ERROR);
```

`DataNotFoundException` deveria retornar 404, `BadRequestException` deveria retornar 400.
Retornar 500 para tudo quebra clientes que dependem do status HTTP para tratar erros.

---

### AP-16 · Response body sem estrutura padronizada 

```java
// ❌ Retorna String plain text — sem campo de código de erro, sem timestamp
return new ResponseEntity<>(errorMessage, HttpStatus.INTERNAL_SERVER_ERROR);
```

APIs REST deveriam retornar um body estruturado e consistente:

```java
// ✅
record ErrorResponse(String code, String message, Instant timestamp) {}
```

##  `RegistrationManager.java` — 5 anti-patterns

### AP-17 · `Long.parseLong` sem tratamento de erro 

```java
// ❌ NumberFormatException não tratada — 500 se id não for numérico
String idStr = request.getParameter("id");
Long id = Long.parseLong(idStr);
```

Se `id = "abc"` ou `id = null`, o serviço lança `NumberFormatException` não tratada.

---

### AP-18 · `repository.findById(id).get()` sem verificação 

```java
// ❌ NoSuchElementException se o cliente não existir
return repository.findById(id).get();
```

`.get()` em um `Optional` vazio lança `NoSuchElementException` — que vira 500.
O correto é usar `.orElseThrow()` com exceção semântica:

```java
// ✅
return repository.findById(id)
    .orElseThrow(() -> new DataNotFoundException("Customer not found: " + id));
```

---

### AP-19 · `System.out.println` em vez de logger 

```java
// ❌ Sem nível de log, sem contexto, sem timestamp, não vai para o log centralizado
System.out.println("ID is a valid CNPJ (for some reason)");
```

Em produção, `System.out` mistura com outros outputs e não é capturado por ferramentas
de observabilidade como CloudWatch ou Datadog:

```java
// ✅
log.debug("ID is a valid CNPJ for id={}", idStr);
```

---

### AP-20 · `Manager` bypassa o `Service` e acessa o `Repository` diretamente 

```java
// ❌ Manager injeta Repository e Service — responsabilidades sobrepostas
private final CustomerRepository repository; // deveria ser só do Service
private final RegistrationService service;   // existe mas não é usado para busca

return repository.findById(id).get(); // Manager faz o que o Service deveria fazer
```

A camada `Service` existe para centralizar regras de negócio e acesso ao banco.
`Manager` chamar `repository` diretamente bypassa essa camada.

---

### AP-21 · Lógica morta — resultado de `validateCnpj` ignorado 

```java
// ❌ Resultado do if nunca usado — código sem propósito
if (tools.validateCnpj(idStr)) {
    System.out.println("ID is a valid CNPJ (for some reason)");
}
```

O comentário `"for some reason"` no próprio código indica que o autor não sabe
por que isso está lá. Dead code deve ser removido.

---

## `RegistrationController.java` — 5 anti-patterns

### AP-22 · `@Autowired` em field (field injection) 

```java
// ❌ Field injection — dificulta testes e esconde dependências
@Autowired
private RegistrationManager manager;
```

Com field injection, não é possível instanciar o controller em testes unitários
sem subir o contexto Spring. O correto é constructor injection:

```java
// ✅
private final RegistrationManager manager;

public RegistrationController(RegistrationManager manager) {
    this.manager = manager;
}
```

---

### AP-23 · `@GetMapping` para operação que usa `HttpServletRequest` 

```java
// ❌ Parâmetro passado via HttpServletRequest em GET — sem validação, sem documentação
@GetMapping("/get-customer")
public Customer getCustomer(HttpServletRequest request) {
    return manager.findCustomer(request);
}
```

O correto é usar `@PathVariable` ou `@RequestParam` com Bean Validation:

```java
// ✅
@GetMapping("/customers/{id}")
public CustomerResponse getCustomer(@PathVariable @Positive Long id) { ... }
```

---

### AP-24 · Retorna entidade JPA diretamente — expõe `password` 

```java
// ❌ Campo password da entidade vai direto no JSON do response
public Customer getCustomer(HttpServletRequest request) {
    return manager.findCustomer(request);
}
```

A entidade `Customer` tem o campo `password` — ele será serializado e enviado
ao cliente. Sempre use um DTO de response:

```java
// ✅
record CustomerResponse(Long id, String name, String email) {}
```

---

### AP-25 · Sem versionamento de endpoint 

```java
// ❌ Sem versão — impossível evoluir a API sem quebrar clientes
@GetMapping("/get-customer")
```

```java
// ✅
@RequestMapping("/v1/customers")
```

---

### AP-26 · `javax.servlet` em vez de `jakarta.servlet` 

```java
// ❌ Namespace antigo — não compila com Spring Boot 3.x
import javax.servlet.http.HttpServletRequest;
```

Spring Boot 3.x usa Jakarta EE 9+ — o namespace mudou de `javax.*` para `jakarta.*`:

```java
// ✅
import jakarta.servlet.http.HttpServletRequest;
```

## `Customer.java` — 5 anti-patterns

### AP-27 · `javax.persistence` em vez de `jakarta.persistence` 

```java
// ❌ Não compila com Spring Boot 3.x
import javax.persistence.Entity;
import javax.persistence.Id;
```

```java
// ✅
import jakarta.persistence.Entity;
import jakarta.persistence.Id;
```

---

### AP-28 · `@Id` sem `@GeneratedValue` — ID nunca gerado 

```java
// ❌ ID precisa ser setado manualmente — qualquer save() sem id falha
@Id
private Long id;
```

```java
// ✅
@Id
@GeneratedValue(strategy = GenerationType.IDENTITY)
private Long id;
```

---

### AP-29 · `password` armazenado sem hash 

```java
// ❌ Senha em plain text no banco — violação gravíssima de segurança
private String password;
```

Senhas devem ser armazenadas com BCrypt ou Argon2, nunca em plain text.

---

### AP-30 · Entidade no mesmo pacote do Controller 

```java
package com.shipay.challenge.endpoints.registration;
// Customer, CustomerRepository, RegistrationController, RegistrationManager, RegistrationService
// todos no mesmo pacote — sem separação de camadas
```

A entidade deveria estar em `domain.entity` ou `infrastructure.persistence`,
não junto com o controller.

---

### AP-31 · Sem constraints de validação na entidade 

```java
// ❌ email pode ser nulo, duplicado, sem formato válido
private String email;
private String password;
```

```java
// ✅
@Column(nullable = false, unique = true)
@Email
private String email;
```

## `RegistrationService.java` — 3 anti-patterns

### AP-32 · `@Autowired` em field (field injection) 

Mesmo problema do AP-22 — dificulta testes e viola a recomendação oficial do Spring.

---

### AP-33 · `orElse(null)` — retorna `null` explicitamente 

```java
// ❌ Propaga null para o caller — NullPointerException silenciosa
public Customer getCustomerById(Long id) {
    return repository.findById(id).orElse(null);
}
```

```java
// ✅
public Customer getCustomerById(Long id) {
    return repository.findById(id)
        .orElseThrow(() -> new DataNotFoundException("Customer not found: " + id));
}
```

### AP-34 · `Service` e `Manager` duplicam acesso ao `Repository` 

`RegistrationService.getCustomerById()` e `RegistrationManager.findCustomer()` fazem
a mesma busca por id diretamente no repository — responsabilidades sobrepostas sem
critério claro de qual camada faz o quê.

##  `Exceptions.java` — 1 anti-pattern

### AP-35 · Múltiplas classes em um único arquivo 

```java
// ❌ BadRequestException, DataNotFoundException e ExternalServiceException
// todas no mesmo arquivo — viola a convenção Java (1 classe pública = 1 arquivo)
public class Exceptions {
    public static class BadRequestException extends RuntimeException { ... }
    public static class DataNotFoundException extends RuntimeException { ... }
    public static class ExternalServiceException extends RuntimeException { ... }
}
```

Além de violar a convenção, o uso fica verboso: `Exceptions.DataNotFoundException`
em vez de apenas `DataNotFoundException`. Cada exceção deveria ter seu próprio arquivo.

##  `DatabaseConfig.java` — 1 anti-pattern

```java
// ❌ Classe de configuração sem nenhum conteúdo
@Configuration
public class DatabaseConfig {
}
```
Se não tem conteúdo, o arquivo deve ser removido.
