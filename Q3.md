# 🧪 Questão 3 — Testes do Event Scheduler


---

## O que precisa ser garantido

```
POST /v1/render/scheduler
   │
   ├── Agendador armazena evento no Redis
   │
   └── Scheduler poll → publica no Kafka → remove do Redis
```

| Premissa | Valor |
|---|---|
| Throughput mínimo | 1.000 requisições/segundo |
| Latência P99 | ≤ 30ms |
| Resiliência | Redis fora do ar não derruba o serviço |
| Consistência | Evento agendado **sempre** chega ao Kafka |
| Sem duplicatas | Evento publicado **uma única vez** |

## Testes Unitários JUnit 5 + Mockito

Validam cada peça isoladamente, sem Redis nem Kafka reais.

### 1.1 Agendamento `SchedulerControllerTest`

```java
@ExtendWith(MockitoExtension.class)
class SchedulerControllerTest {

    @Mock
    private JobScheduler jobScheduler;

    @InjectMocks
    private SchedulerController controller;

    @Test
    @DisplayName("deve retornar job_id quando agendamento é bem sucedido")
    void shouldReturnJobIdOnSuccess() {
        var request = new SchedulerRequest(
                LocalDateTime.now().plusMinutes(5),
                "{\"videoId\": \"abc-123\"}"
        );
        when(jobScheduler.enqueueAt(any(), eq("publish_event"), any()))
                .thenReturn("job-uuid-001");

        var response = controller.scheduler(request);

        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.OK);
        assertThat(response.getBody()).containsEntry("job_id", "job-uuid-001");
    }

    @Test
    @DisplayName("deve retornar 400 quando payload é nulo")
    void shouldReturn400WhenPayloadIsNull() {
        assertThatThrownBy(() -> controller.scheduler(null))
                .isInstanceOf(ResponseStatusException.class)
                .hasMessageContaining("where is the request payload?");
    }

    @Test
    @DisplayName("deve retornar 500 quando JobScheduler lança exceção")
    void shouldReturn500WhenSchedulerFails() {
        var request = new SchedulerRequest(LocalDateTime.now().plusMinutes(1), "{}");
        when(jobScheduler.enqueueAt(any(), any(), any()))
                .thenThrow(new RuntimeException("Redis connection refused"));

        assertThatThrownBy(() -> controller.scheduler(request))
                .isInstanceOf(ResponseStatusException.class)
                .hasMessageContaining("Error during scheduler event");
    }
}
```

> 💡 **Por que `@Mock` e não `@MockitoBean`?** Esse teste não sobe o contexto Spring
> o controller é instanciado diretamente via `@InjectMocks`. `@MockitoBean` só é necessário
> quando o teste usa `@SpringBootTest` ou `@WebMvcTest`.

### 1.2 Poll e Publicação `EventSchedulerServiceTest`

Usa `@BDDMockito` para alinhar o código com o padrão **Given / When / Then** dos `@DisplayName`,
e fixtures privadas para eliminar repetição na criação dos objetos:

```java
@ExtendWith(MockitoExtension.class)
class EventSchedulerServiceTest {

    @Mock private RedisJobRepository redisJobRepository;
    @Mock private KafkaTemplate<String, String> kafkaTemplate;

    @InjectMocks
    private EventSchedulerService schedulerService;

    // --- fixtures ---

    private ScheduledJob expiredJob(String id) {
        return ScheduledJob.builder()
                .id(id).topic("status-check-topic")
                .content("{\"videoId\":\"xyz\"}")
                .scheduledAt(LocalDateTime.now().minusSeconds(1))
                .build();
    }

    private ScheduledJob futureJob(String id) {
        return ScheduledJob.builder()
                .id(id).topic("status-check-topic").content("{}")
                .scheduledAt(LocalDateTime.now().plusMinutes(10))
                .build();
    }

    // --- testes ---

    @Test
    @DisplayName("deve publicar no Kafka e remover do Redis quando evento vence")
    void shouldPublishAndRemoveExpiredJob() {
        given(redisJobRepository.findExpiredJobs(any()))
                .willReturn(List.of(expiredJob("job-001")));

        schedulerService.pollAndPublish();

        then(kafkaTemplate).should().send("status-check-topic", "{\"videoId\":\"xyz\"}");
        then(redisJobRepository).should().delete("job-001");
    }

    @Test
    @DisplayName("não deve publicar eventos que ainda não venceram")
    void shouldNotPublishFutureJobs() {
        given(redisJobRepository.findExpiredJobs(any())).willReturn(Collections.emptyList());

        schedulerService.pollAndPublish();

        then(kafkaTemplate).shouldHaveNoInteractions();
    }

    @Test
    @DisplayName("deve remover do Redis SOMENTE após confirmação do Kafka")
    void shouldNotRemoveFromRedisWhenKafkaFails() {
        given(redisJobRepository.findExpiredJobs(any()))
                .willReturn(List.of(expiredJob("job-003")));
        given(kafkaTemplate.send(any(), any()))
                .willThrow(new RuntimeException("Kafka unavailable"));

        schedulerService.pollAndPublish();

        then(redisJobRepository).should(never()).delete("job-003");
    }
}
```

> ⚠️ **O teste `shouldNotRemoveFromRedisWhenKafkaFails` é crítico.** Garante que o evento
> não é perdido se o Kafka estiver fora do ar o job permanece no Redis para ser
> reprocessado no próximo ciclo do scheduler.

> 💡 **Por que `@BDDMockito`?** O estilo `given/then` espelha o `@DisplayName` de cada teste
> quem lê o código entende o cenário sem precisar de comentários. Já vem no Mockito,
> sem dependência extra.

### 2.1 Fluxo completo — agendamento → Kafka

Usa `@ServiceConnection` (Spring Boot 3.1+) para eliminar o `@DynamicPropertySource` manual,
`@BeforeEach` para isolamento entre testes e fixtures para evitar cast inline frágil:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.NONE)
@Testcontainers
class EventSchedulerIntegrationTest {

    @Container @ServiceConnection
    static RedisContainer redis = new RedisContainer("redis:7-alpine");

    @Container @ServiceConnection
    static KafkaContainer kafka = new KafkaContainer(
            DockerImageName.parse("apache/kafka:3.8.0"));

    @Autowired private SchedulerController controller;
    @Autowired private EventSchedulerService schedulerService;
    @Autowired private RedisJobRepository redisJobRepository;

    @BeforeEach
    void cleanUp() {
        redisJobRepository.deleteAll(); // isolamento entre testes
    }

    // --- fixtures ---

    private SchedulerRequest scheduleNow(String videoId) {
        return new SchedulerRequest(
                LocalDateTime.now(),
                "{\"videoId\": \"%s\"}".formatted(videoId)
        );
    }

    private String scheduleAndGetJobId(String videoId) {
        var response = controller.scheduler(scheduleNow(videoId));
        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.OK);
        var body = (Map<?, ?>) response.getBody();
        assertThat(body).containsKey("job_id");
        return (String) body.get("job_id");
    }

    private List<String> consumeFromKafka(String topic) {
        consumer.subscribe(List.of(topic));
        return StreamSupport
                .stream(consumer.poll(Duration.ofSeconds(5)).spliterator(), false)
                .map(ConsumerRecord::value)
                .toList();
    }

    // --- testes ---

    @Test
    @DisplayName("evento agendado deve chegar ao Kafka após o scheduler executar")
    void scheduledEventShouldReachKafka() {
        controller.scheduler(scheduleNow("integration-test-001"));

        await().atMost(5, SECONDS).untilAsserted(() -> {
            schedulerService.pollAndPublish();
            assertThat(consumeFromKafka("status-check-topic"))
                    .contains("integration-test-001");
        });
    }

    @Test
    @DisplayName("evento deve ser removido do Redis após publicação no Kafka")
    void shouldRemoveFromRedisAfterKafkaPublish() {
        var jobId = scheduleAndGetJobId("cleanup-test");

        schedulerService.pollAndPublish();

        assertThat(redisJobRepository.findById(jobId)).isEmpty();
    }

    @Test
    @DisplayName("evento NÃO deve ser duplicado se o scheduler executar duas vezes")
    void shouldNotPublishDuplicateEvents() {
        controller.scheduler(scheduleNow("dedup-test"));

        schedulerService.pollAndPublish();
        schedulerService.pollAndPublish();

        assertThat(consumeFromKafka("status-check-topic")).hasSize(1);
    }
}
```

> 💡 **Por que `@ServiceConnection`?** Elimina o `@DynamicPropertySource` manual o Spring Boot
> detecta o tipo do container e configura host, porta e credenciais automaticamente.

> 💡 **Por que `webEnvironment = NONE`?** Esse teste valida a lógica de agendamento e
> persistência — não precisa subir o servidor HTTP. Reduz o tempo de startup do contexto Spring.

### 2.2 Resiliência — Redis indisponível

Isolado em classe própria com `@DirtiesContext` para não contaminar os outros testes:

```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.NONE)
@Testcontainers
@DirtiesContext // recria o contexto Spring após essa classe
class SchedulerResilienceTest {

    @Container @ServiceConnection
    static RedisContainer redis = new RedisContainer("redis:7-alpine");

    @Autowired
    private SchedulerController controller;

    @Test
    @DisplayName("deve retornar 500 com mensagem clara quando Redis está indisponível")
    void shouldReturn500WhenRedisIsDown() {
        redis.stop();

        var request = new SchedulerRequest(LocalDateTime.now().plusMinutes(1), "{}");

        assertThatThrownBy(() -> controller.scheduler(request))
                .isInstanceOf(ResponseStatusException.class)
                .satisfies(ex -> {
                    var e = (ResponseStatusException) ex;
                    assertThat(e.getStatusCode())
                            .isEqualTo(HttpStatus.INTERNAL_SERVER_ERROR);
                    assertThat(e.getReason())
                            .contains("Error during scheduler event");
                });
    }
}
```

> ⚠️ **Por que classe separada com `@DirtiesContext`?** `redis.stop()` para o container
> para toda a JVM se estivesse na mesma classe, todos os testes seguintes falhariam
> por falta de Redis. O `@DirtiesContext` garante que o contexto Spring seja recriado
> após essa classe, sem afetar os demais.

> ⚠️ **Por que `.satisfies()` em vez de `.extracting("status")`?** Acesso por String é
> frágil quebra silenciosamente se o campo mudar de nome. O `.satisfies()` com cast
> tipado verifica tanto o código HTTP quanto a mensagem de erro em um único bloco legível.

## Testes de Carga Gatling (Java DSL)

Validam as premissas de **1.000 req/s** e **P99 ≤ 30ms**.

### Dependência (`pom.xml`)

```xml
<dependency>
    <groupId>io.gatling.highcharts</groupId>
    <artifactId>gatling-charts-highcharts</artifactId>
    <version>3.9.5</version>
    <scope>test</scope>
</dependency>
```

### `SchedulerLoadTest.java`

```java
public class SchedulerLoadTest extends Simulation {

    // --- 1. Feeder — gera dados únicos por usuário virtual ---
    // Evita que 1.000 req/s enviem o mesmo payload, simulando produção real
    private final Iterator<Map<String, Object>> feeder =
        Stream.generate((Supplier<Map<String, Object>>) () -> Map.of(
            "videoId",   UUID.randomUUID().toString(),
            "datetime",  LocalDateTime.now().plusSeconds(30).toString()
        )).iterator();

    // --- 2. Protocolo HTTP base ---
    private final HttpProtocolBuilder httpProtocol = http
        .baseUrl("http://localhost:8080")
        .header("Authorization", "Bearer #{token}")
        .header("Content-Type", "application/json")
        .disableWarmUp(); // sem aquecimento — mede latência real desde o início

    // --- 3. Cenário — o que cada usuário virtual faz ---
    private final ScenarioBuilder scn = scenario("Scheduler Load Test")
        .feed(feeder)           // injeta videoId e datetime únicos por usuário
        .exec(
            http("POST /v1/render/scheduler")
                .post("/v1/render/scheduler")
                .body(StringBody(
                    "{"schedulerDatetime": "#{datetime}"," +
                    " "eventContent": "{\"videoId\": \"#{videoId}\"}"}"
                ))
                .check(status().is(200))
                .check(jsonPath("$.job_id").exists()) // valida que job_id veio no response
        );

    // --- 4. Setup — estratégia de injeção de carga ---
    {
        setUp(
            scn.injectOpen(
                rampUsersPerSec(100).to(1000).during(30), // aquece: 100 → 1.000 req/s em 30s
                constantUsersPerSec(1000).during(60)       // sustenta 1.000 req/s por 60s
            ).protocols(httpProtocol)
        )
        // throttle — proteção extra: limita o teto mesmo que a injeção dispare mais
        .throttle(
            reachRps(1000).in(30),   // sobe para 1.000 req/s em 30s
            holdFor(60)              // mantém por 60s
        )
        // --- 5. Assertions — falham o build se as premissas não forem atingidas ---
        .assertions(
            global().responseTime().percentile(99).lte(30),    // P99 ≤ 30ms ✅
            global().successfulRequests().percent().gte(99.9), // 99.9% de sucesso
            global().requestsPerSec().gte(1000.0)              // throughput mínimo
        );
    }
}


```

### O que cada parte faz

**Feeder**: Gera dados dinâmicos por usuário virtual via `Stream.generate()`, evitando que todos os 1.000 req/s enviem o mesmo payload. Sem o feeder, o Redis poderia cachear o mesmo job e distorcer os resultados.

**`rampUsersPerSec` + `constantUsersPerSec`**: A rampa evita spike inicial que derruba o serviço antes do teste começar. A fase constante é onde P99 e throughput são medidos de verdade.

**`throttle`**: Camada extra de proteção que limita o teto de req/s independente da injeção. Se a injeção disparar mais usuários do que esperado, o throttle segura em 1.000 req/s.

**`jsonPath("$.job_id").exists()`**: Além do status 200, valida que o `job_id` realmente veio no response. Um status 200 com body vazio ou malformado passaria no check de status mas falharia aqui.

### Interpretando os resultados

```
---- Requests -------------------------------------------------------
> POST /v1/render/scheduler                         OK=60000  KO=0
---- Response Time Distribution -------------------------------------
  t < 10ms          45230 ( 75%)
  10ms <= t < 20ms  12100 ( 20%)
  20ms <= t < 30ms   2400 (  4%)
  t >= 30ms           270 (  0.45%)  ← P99 dentro do limite ✅
---- Assertions ------------------------------------------------------
  global.responseTime.percentile99 is lte 30ms          PASSED ✅
  global.successfulRequests.percent is gte 99.9%         PASSED ✅
  global.requestsPerSec is gte 1000                      PASSED ✅
```

> ⚠️ **P99 ≠ média.** P99 de 30ms significa que 99% das requisições responderam em
> até 30ms o 1% restante pode ultrapassar sem violar a premissa. O que **não** pode
> acontecer é o P99 ultrapassar 30ms sob carga sustentada de 1.000 req/s.

> ⚠️ **Se o P99 falhar:** os primeiros suspeitos são o pool de conexões do Redis
> (`spring.data.redis.lettuce.pool.max-active`) e o número de threads do Tomcat
> (`server.tomcat.threads.max`). Com Virtual Threads no Java 21, o Tomcat pode ser
> configurado para usar virtual threads e eliminar o gargalo de threads.

> ⚠️ https://docs.gatling.io/concepts/injection/
