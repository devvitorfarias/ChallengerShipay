# 📊 Questão 2 — Relatório de Lançamentos por Cliente

---

## 🎯 O que faz

Registra cada lançamento de foguete em uma **base de dados paralela** para geração de relatórios
por cliente em intervalos de 30 dias — **sem nenhum impacto no fluxo principal de lançamento**.

Se o registro do relatório falhar por qualquer motivo, o lançamento já foi concluído e o cliente
já recebeu a resposta. O erro é apenas logado.

---

## Arquitetura

```
Cliente
   │
   │  POST /v1/rocket/launch
   ▼
LaunchController
   │
   ├── rulesEngine.isLaunchApproved()
   ├── service.preFlightCheck()
   ├── service.countdown()
   ├── service.launch() ──────────────────▶ banco principal (porta 5432)
   │        │
   │   ResponseEntity.ok(response) ◀───────── cliente já recebeu ✅
   │
   └── reportService.recordAsync()
              │
              ▼ (virtual thread separada — não bloqueia)
        ReportService
              │
              ▼
        banco relatório (porta 5433)
              │
         falhou? → só loga ✅ (não propaga)
```

> **Zero impacto garantido.** O `return ResponseEntity.ok(response)` executa antes
> do banco de relatório ser tocado. O `@Async` dispara a gravação em uma virtual
> thread separada após a resposta já ter sido enviada ao cliente.

---

## Virtual Threads + `@Async` (Java 21)

O executor de relatório usa virtual threads sem pool fixo, sem tuning manual:

```java
// AsyncConfig.java
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean(name = "reportExecutor")
    public Executor reportExecutor() {
        // Uma virtual thread por tarefa — custo ~1KB vs ~1MB de platform thread
        return Executors.newVirtualThreadPerTaskExecutor();
    }
}
```

> **Por que Virtual Threads aqui?** Gravar no banco é uma operação de I/O  a thread
> passa a maior parte do tempo esperando o PostgreSQL confirmar o `INSERT`. Virtual threads
> liberam a platform thread durante essa espera, sem custo de bloqueio real.

## 🔌 Ponto de captura  `LaunchController`

O registro é disparado **após** o lançamento ser concluído com sucesso, sem alterar
nenhuma lógica existente do controller:

```java
@PostMapping("/launch")
@PreAuthorize("@authService.hasPermission('resource', '/v1/rocket/launch/post')")
public ResponseEntity<?> launch(
        @RequestHeader(value = "trace_id", required = false) String traceId,
        @RequestBody LaunchRequestBody schema) {

    try {
        if (traceId == null) traceId = UUID.randomUUID().toString();

        if (schema == null)
            throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "where is the request payload?");

        if (!rulesEngine.isLaunchApproved(schema))
            throw new ResponseStatusException(HttpStatus.METHOD_NOT_ALLOWED, "your launch is not allowed.");

        PreFlightStatus preFlight = service.preFlightCheck();
        if (!rulesEngine.isPreFlightStatusOk(preFlight.getStatus()))
            throw new ResponseStatusException(HttpStatus.INTERNAL_SERVER_ERROR, "your launch is compromised, please abort.");

        CountdownStatus countdownStatus = service.countdown();
        LaunchResponse response = service.launch(traceId, schema.getCustomerId(), countdownStatus, preFlight);

        // ✅ Lançamento concluído — dispara o registro de forma assíncrona
        reportService.recordAsync(traceId, schema.getCustomerId(), response);

        return ResponseEntity.ok(response); // ← cliente recebe aqui, independente do relatório

    } catch (ResponseStatusException e) {
        throw e;
    } catch (Exception e) {
        throw new ResponseStatusException(HttpStatus.INTERNAL_SERVER_ERROR, "Error during launch... " + e.getMessage());
    }
}
```

## 📝 ReportService: isolado e blindado

```java
// ReportService.java
@Service
@Slf4j
@RequiredArgsConstructor
public class ReportService {

    private final ReportRepository reportRepository;

    @Async("reportExecutor")
    public void recordAsync(String traceId, String customerId, LaunchResponse response) {
        try {
            var record = LaunchReportRecord.builder()
                .traceId(traceId)
                .customerId(customerId)
                .launchedAt(LocalDateTime.now())
                .status(response.getStatus())
                .build();

            reportRepository.save(record);
            log.info("Report recorded for customerId={} traceId={}", customerId, traceId);

        } catch (Exception e) {
            // Falha aqui NUNCA propaga para o LaunchController
            log.error("Report recording failed — customerId={} traceId={}: {}",
                customerId, traceId, e.getMessage());
        }
    }
}
```

> ⚠️ **O `try/catch` interno é intencional.** Sem ele, uma exceção não capturada
> em um método `@Async` seria engolida silenciosamente pelo Spring ou propagada
> de forma imprevisível. O `catch` garante que o erro sempre seja logado e nunca
> afete o caller.

### Por que `@Async` + `try/catch` é a melhor estratégia aqui?

O enunciado tem dois requisitos que criam uma tensão:

```
1. Gravar dados para relatório
2. Falha na gravação NÃO pode afetar o lançamento
```

São duas camadas de proteção com responsabilidades distintas:

**`@Async` resolve o isolamento de thread:**

```
LaunchController thread          ReportService virtual thread
        │                                    │
  service.launch() ✅                        │
        │                                    │
  recordAsync() ──── dispara ──────────────▶ │
        │            e esquece               │ reportRepository.save()
        │                                    │
  return ResponseEntity ✅        falhou? só loga ✅
```

**`try/catch` interno resolve o que `@Async` sozinho não resolve:**

Sem o `try/catch`, uma exceção em um método `@Async` vai para o `AsyncUncaughtExceptionHandler`
padrão do Spring que por padrão apenas loga, sem nenhum controle da sua parte.
Com o `catch` explícito você decide o que acontece com o erro:

```java
} catch (Exception e) {
    // aqui você controla:
    // - só logar (atual)
    // - enviar para uma fila de dead letter
    // - incrementar métricas
    // - disparar um alerta
    log.error("Report recording failed — customerId={} traceId={}: {}",
        customerId, traceId, e.getMessage());
}
```

> 💡 **Evolução natural para produção:** substituir o `recordAsync` por um evento Kafka
> (`kafkaTemplate.send("launch-report-topic", event)`). Isso garante que mesmo se o banco
> de relatório ficar fora do ar por horas, nenhum evento é perdido. Para o escopo deste
> challenge, `@Async` + `try/catch` entrega o requisito com complexidade mínima.

## 🗄️ Base de Dados Paralela

O `ReportRepository` usa o datasource secundário já configurado no projeto 
completamente isolado do banco principal:

```java
// ReportDatasourceConfig.java (já existe no projeto)
@Configuration
@Profile("!test")
public class ReportDatasourceConfig {

    @Bean(name = "reportDataSource")
    @ConfigurationProperties("spring.datasource.report")
    public DataSource reportDataSource() {
        return DataSourceBuilder.create().build();
    }
}
```

```yaml
# application.yml
spring:
  datasource:
    # Banco principal — lançamentos
    url: jdbc:postgresql://localhost:5432/shipay_db

    report:
      # Banco paralelo — relatórios (porta diferente)
      url: jdbc:postgresql://localhost:5433/shipay_report_db
      username: shipay_report
      password: report123
```

## Entidade do Relatório

```java
// LaunchReportRecord.java
@Entity
@Table(name = "launch_report")
@Builder
@NoArgsConstructor
@AllArgsConstructor
public class LaunchReportRecord {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "trace_id")
    private String traceId;

    @Column(name = "customer_id")
    private String customerId;

    @Column(name = "launched_at")
    private LocalDateTime launchedAt;

    @Column(name = "status")
    private String status;
}
```

## Query do Relatório 30 dias por cliente

```java
// ReportRepository.java
public interface ReportRepository extends JpaRepository<LaunchReportRecord, Long> {

    @Query("""
                SELECT r.customerId, COUNT(r) as totalLaunches
                FROM LaunchReportRecord r
                WHERE r.launchedAt >= :from AND r.launchedAt <= :to
                GROUP BY r.customerId
                ORDER BY totalLaunches DESC
            """)
    List<CustomerLaunchSummary> findByPeriod(
            @Param("from") LocalDateTime from,
            @Param("to") LocalDateTime to
    );
}
```

```java
// Chamada para gerar o relatório dos últimos 30 dias:
var report = reportRepository.findByPeriod(
                LocalDateTime.now().minusDays(30),
                LocalDateTime.now()
        );
```
