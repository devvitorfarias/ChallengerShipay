# 🔍 Code Review — `UserExportBot.java`

> Revisão técnica do robô de exportação de dados da tabela `users` para `.xlsx`.

---

### 1. Credenciais hardcoded no código-fonte

**O que significa:** Usuário e senha do banco de dados estão escritos literalmente dentro do arquivo `.java`. Qualquer pessoa com acesso ao repositório Git — inclusive o histórico de commits — consegue ver as credenciais de produção.

**Trecho problemático:**
```java
Connection conn = DriverManager.getConnection(
    "jdbc:postgresql://127.0.0.1:5432/bot_db",
    "postgres",
    "123mudar"  // ← senha exposta no repositório!
);
```

**Causa raiz:** Falta de externalização de configurações sensíveis.

**Solução recomendada:** Usar variáveis de ambiente. Elas são injetadas pelo sistema operacional ou orquestrador (Docker, Kubernetes) em tempo de execução, sem aparecer no código ou no repositório.

```java
// ✅ Correto
String url  = System.getenv("DB_URL");
String user = System.getenv("DB_USER");
String pass = System.getenv("DB_PASSWORD");
Connection conn = DriverManager.getConnection(url, user, pass);
```

### 2. Senha dos usuários exportada na planilha

**O que significa:** A coluna `Password` (que contém hashes BCrypt das senhas) está sendo incluída na planilha Excel exportada. Isso viola a LGPD (Art. 46) e o GDPR (Art. 32), pois expõe dados de autenticação fora do sistema.

**Trecho problemático:**
```java
header.createCell(3).setCellValue("Password"); // ← coluna de senha na planilha!

System.out.println("Password: " + rs.getString(4)); // ← imprime hash no console!
row.createCell(3).setCellValue(rs.getString(4));     // ← grava hash na planilha!
```

**Causa raiz:** Ausência de filtro sobre quais dados podem ser exportados. Mesmo sendo hash, hashes BCrypt podem ser atacados offline com GPUs modernas caso o arquivo vaze.

**Solução recomendada:** Remover completamente a coluna `Password` da exportação. A planilha deve conter apenas dados operacionais necessários.

```java
// ✅ Correto — exportar apenas o necessário
header.createCell(0).setCellValue("Id");
header.createCell(1).setCellValue("Name");
header.createCell(2).setCellValue("Email");
header.createCell(3).setCellValue("Role Id");
header.createCell(4).setCellValue("Created At");
// Password removida intencionalmente
```

### 3. Dados pessoais (PII) impressos no console

**O que significa:** PII (_Personally Identifiable Information_) são dados que identificam uma pessoa como nome, e-mail e senha. Imprimi-los no console é perigoso porque logs de console são frequentemente coletados por sistemas de monitoramento (Splunk, Datadog, ELK), onde ficam armazenados e acessíveis por terceiros.

**Trecho problemático:**
```java
System.out.println("Name: "     + rs.getString(2)); // ← PII no log!
System.out.println("Email: "    + rs.getString(3)); // ← PII no log!
System.out.println("Password: " + rs.getString(4)); // ← crítico!
```

**Causa raiz:** Falta de política de mascaramento de dados sensíveis nos logs.

**Solução recomendada:** Logar apenas metadados não sensíveis, como contadores e timestamps.

```java
// ✅ Correto — apenas metadados, sem dados pessoais
logger.info("Processando registro {} de {}", index, totalRows);
```

### 4. Recursos não fechados em caso de exceção (Resource Leak)

**O que significa:** `Connection`, `Statement`, `ResultSet` e `Workbook` precisam ser explicitamente fechados após o uso. No código atual, se qualquer linha entre a abertura e o `close()` lançar uma exceção, os recursos ficam abertos para sempre. Com o tempo, o PostgreSQL atinge o limite de conexões simultâneas e começa a recusar novas conexões.

**Trecho problemático:**
```java
Connection conn = DriverManager.getConnection(...);
Statement stmt  = conn.createStatement();
// ← se algo falhar aqui, os recursos abaixo nunca são executados
fileOut.close();
workbook.close();
conn.close(); // ← nunca alcançado!
```

**Causa raiz:** Gerenciamento manual de recursos sem garantia de fechamento.

**Solução recomendada:** `try-with-resources` (Java 7+). A JVM chama `close()` automaticamente ao sair do bloco, mesmo que uma exceção ocorra — porque essas classes implementam a interface `AutoCloseable`.

```java
// ✅ Correto — fechamento garantido em qualquer cenário
try (Connection conn      = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql);
     ResultSet rs         = ps.executeQuery();
     Workbook workbook    = new XSSFWorkbook();
     FileOutputStream out = new FileOutputStream(fileName)) {

    // recursos fechados automaticamente aqui
}
```

### 5. `SELECT *` sem filtro nem paginação

**O que significa:** `SELECT * FROM users` busca todas as colunas de todos os registros de uma vez e os carrega na memória RAM. Com milhões de usuários isso causa `OutOfMemoryError` na JVM, realiza um _full table scan_ no PostgreSQL (lento e pesado) e pode gerar uma planilha Excel acima do limite de ~1 milhão de linhas.

**Trecho problemático:**
```java
ResultSet rs = stmt.executeQuery("SELECT * FROM users;");
// ← carrega TODA a tabela na memória de uma vez!
```

**Causa raiz:** Ausência de planejamento de capacidade e paginação.

**Solução recomendada:** Selecionar apenas as colunas necessárias e processar em lotes com `LIMIT` e `OFFSET`, mantendo consumo de memória constante independente do tamanho da tabela.

```java
// ✅ Correto — colunas explícitas + paginação
String sql = """
    SELECT id, name, email, role_id, created_at
    FROM users
    ORDER BY id
    LIMIT ? OFFSET ?
    """;
PreparedStatement ps = conn.prepareStatement(sql);
ps.setInt(1, PAGE_SIZE);  // ex: 10.000 por lote
ps.setInt(2, offset);
```

### 6. `FileOutputStream` não fechado em caso de falha

**O que significa:** Assim como o problema 4, se `workbook.write(fileOut)` lançar uma exceção, o `fileOut.close()` nunca é chamado. O arquivo fica bloqueado pelo processo e pode ficar corrompido ou inacessível.

**Trecho problemático:**
```java
FileOutputStream fileOut = new FileOutputStream(fileName);
workbook.write(fileOut);
fileOut.close(); // ← não executado se write() falhar!
```

**Causa raiz:** Mesma raiz do problema 4 — gerenciamento manual sem garantia de fechamento.

**Solução recomendada:** Incluir o `FileOutputStream` no mesmo bloco `try-with-resources`.

```java
// ✅ Correto
try (FileOutputStream fileOut = new FileOutputStream(fileName)) {
    workbook.write(fileOut);
} // fechado automaticamente aqui
```
### 7. Acesso a colunas por índice numérico

**O que significa:** `rs.getLong(5)` e `rs.getString(6)` são "números mágicos" — ninguém sabe o que é a coluna 5 sem contar na query. Se a query mudar de `SELECT *` para colunas explícitas, os índices mudam silenciosamente, causando bugs difíceis de detectar (ex: `role_id` recebendo o valor de `created_at`).

**Trecho problemático:**
```java
row.createCell(4).setCellValue(rs.getLong(5));   // ← o que é a coluna 5?
row.createCell(5).setCellValue(rs.getString(6)); // ← o que é a coluna 6?
```

**Causa raiz:** Uso de `SELECT *` combinado com acesso por posição em vez de nome.

**Solução recomendada:** Acessar colunas pelo nome — seguro, legível e imune a reordenações na query.

```java
// ✅ Correto — autodocumentado e seguro
row.createCell(3).setCellValue(rs.getLong("role_id"));
row.createCell(4).setCellValue(rs.getString("created_at"));
```

### 8. `Statement` em vez de `PreparedStatement`

**O que significa:** `Statement` executa SQL como texto puro. `PreparedStatement` pré-compila a query no banco, usa parâmetros tipados com `?` (prevenindo SQL Injection) e tem melhor performance em execuções repetidas. É a prática padrão mesmo para queries sem parâmetros dinâmicos.

**Trecho problemático:**
```java
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM users;");
```

**Causa raiz:** Uso da API JDBC mais básica sem considerar boas práticas.

**Solução recomendada:**

```java
// ✅ Correto
PreparedStatement ps = conn.prepareStatement(
    "SELECT id, name, email, role_id, created_at FROM users ORDER BY id"
);
ResultSet rs = ps.executeQuery();
```

### 9. Nova conexão JDBC criada a cada iteração do loop

**O que significa:** Abrir uma conexão JDBC envolve handshake TCP + autenticação no PostgreSQL — é uma operação cara. No código atual isso acontece a cada 60 segundos, desnecessariamente.

**Trecho problemático:**
```java
while (true) {
    Connection conn = DriverManager.getConnection(...); // ← nova conexão todo ciclo!
    // ...
    conn.close();
    Thread.sleep(60000);
}
```

**Causa raiz:** Ausência de pool de conexões.

**Solução recomendada:** HikariCP é o pool de conexões mais rápido para Java. Ele mantém um conjunto de conexões abertas e as empresta/devolve instantaneamente, sem o custo de abertura a cada uso.

```java
// ✅ Correto — pool criado uma vez, conexões reutilizadas
HikariConfig config = new HikariConfig();
config.setJdbcUrl(System.getenv("DB_URL"));
HikariDataSource ds = new HikariDataSource(config);

while (true) {
    try (Connection conn = ds.getConnection()) { // conexão do pool
        // ...
    }
}
```

### 10. Exceções capturadas de forma genérica e silenciadas

**O que significa:** `catch(Exception e)` captura absolutamente tudo, inclusive erros de programação como `NullPointerException` que deveriam parar a execução. `e.printStackTrace()` apenas imprime no console, sem nível de log, timestamp ou possibilidade de alertas automáticos.

**Trecho problemático:**
```java
} catch (Exception e) {
    e.printStackTrace(); // ← silencia o erro e continua o loop!
}
```

**Causa raiz:** Tratamento de erro superficial sem distinção do tipo de falha.

**Solução recomendada:** Capturar exceções específicas e usar logger com contexto.

```java
// ✅ Correto — tratamento específico e com contexto
} catch (SQLException e) {
    logger.error("Falha na consulta ao banco na iteração {}: {}", index, e.getMessage(), e);
} catch (IOException e) {
    logger.error("Falha ao gravar arquivo {}: {}", fileName, e.getMessage(), e);
}
```

### 11. `System.out.println` em vez de logger estruturado

**O que significa:** `System.out.println` escreve texto puro no console sem timestamp, sem nível de severidade (INFO/WARN/ERROR) e sem a possibilidade de direcionar saída para arquivos ou sistemas de monitoramento. Em produção isso é inaceitável.

**Solução recomendada:** SLF4J com Logback — padrão da indústria Java.

```java
// ✅ Correto
private static final Logger logger = LoggerFactory.getLogger(UserExportBot.class);
logger.info("Exportação concluída. Arquivo: {} | Registros: {}", fileName, totalRows);
// Saída: 2026-03-09 15:00:01 INFO  UserExportBot - Exportação concluída. Arquivo: data_export_20260309.xlsx | Registros: 4523
```

### 12. Intervalo de execução hardcoded

**O que significa:** `Thread.sleep(60000)` fixa 1 minuto no código. Para alterar o intervalo é necessário modificar o código, recompilar e redesdobrar a aplicação — o que é impraticável em produção.

**Solução recomendada:** Externalizar via variável de ambiente, seguindo o princípio _12-Factor App_ (configuração separada do código).

```java
// ✅ Correto — configurável sem recompilar
long interval = Long.parseLong(
    System.getenv().getOrDefault("EXPORT_INTERVAL_MS", "60000")
);
Thread.sleep(interval);
```

---

### 13. Loop infinito com `Thread.sleep` em vez de scheduler

**O que significa:** `while(true)` + `Thread.sleep()` é a abordagem mais primitiva de agendamento. Se a exportação demorar mais que o intervalo configurado, as execuções se acumulam. Além disso, não há controle de _shutdown_ gracioso — ao parar a aplicação no meio de uma exportação, o arquivo pode ficar corrompido.

**Solução recomendada:** `ScheduledExecutorService` garante o intervalo entre execuções e permite shutdown limpo.

```java
// ✅ Correto — agendamento robusto e controlado
ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor();

Runtime.getRuntime().addShutdownHook(new Thread(scheduler::shutdown)); // shutdown gracioso

scheduler.scheduleAtFixedRate(
    UserExportBot::exportUsers,
    0,          // delay inicial
    60,         // intervalo
    TimeUnit.SECONDS
);
```
