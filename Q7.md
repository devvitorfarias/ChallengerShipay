#Design Patterns — Normalização de Serviços de Terceiros

> Padrões de projeto escolhidos para normalizar provedores de e-mail e SMS em Java 21.

---

## Visão Geral

Ao integrar múltiplos provedores de terceiros (SendGrid, AWS SES, Twilio, etc.), o maior risco é o acoplamento direto o sistema fica amarrado a um fornecedor específico. Se o contrato mudar, o provedor sair do ar ou você quiser trocar, seria necessário alterar código em vários lugares.

A solução é criar uma camada de abstração usando a combinação de 3 padrões:

| Padrão | Papel na solução |
|---|---|
| **Strategy** | Define a interface comum entre todos os provedores |
| **Factory** | Decide qual provedor instanciar em tempo de execução |
| **Adapter** | Traduz a API de cada provedor para a interface comum |

---

## Strategy: O Núcleo da Solução

### O que é

Strategy define uma família de algoritmos intercambiáveis encapsulados por uma interface comum. O código que envia a notificação não sabe nem precisa saber qual provedor está sendo usado.

### Por que usar aqui

Cada provedor de e-mail (SendGrid, AWS SES, Mailgun) tem sua própria API, mas todos fazem a mesma coisa: **enviar uma mensagem**. O Strategy força todos a respeitarem o mesmo contrato.

### Implementação em Java 21

```java
// Contrato comum para todos os provedores de e-mail
public interface EmailProvider {
    void send(EmailMessage message);
}

// Contrato comum para todos os provedores de SMS
public interface SmsProvider {
    void send(SmsMessage message);
}

// Records do Java 21 — imutáveis, sem boilerplate
public record EmailMessage(
    String from,
    String to,
    String subject,
    String body
) {}

public record SmsMessage(
    String from,
    String to,
    String body
) {}
```

```java
// Serviço de notificação — não conhece nenhum provedor específico
@Service
@RequiredArgsConstructor
public class NotificationService {

    private final EmailProvider emailProvider; // ← injetado pelo Spring/Factory
    private final SmsProvider   smsProvider;

    public void sendEmail(EmailMessage message) {
        emailProvider.send(message);
    }

    public void sendSms(SmsMessage message) {
        smsProvider.send(message);
    }
}
```

> **Benefício:** Para trocar de SendGrid para AWS SES, basta trocar a implementação injetada. O `NotificationService` não muda uma linha.

## Adapter: A Tradução entre Mundos

### O que é

Adapter converte a interface incompatível de um terceiro para a interface que o seu sistema espera. É como um adaptador de tomada a voltagem é diferente, mas você consegue usar o aparelho normalmente.

### Por que usar aqui

Cada provedor tem sua própria API proprietária com métodos, parâmetros e modelos de dados completamente diferentes. O Adapter absorve essa diferença e expõe sempre a mesma interface limpa.

### Implementação em Java 21

```java
// Adapter para SendGrid
@Component("sendgrid")
public class SendGridEmailAdapter implements EmailProvider {

    // SDK do SendGrid (terceiro)
    private final SendGrid sendGridClient;

    public SendGridEmailAdapter(@Value("${sendgrid.api-key}") String apiKey) {
        this.sendGridClient = new SendGrid(apiKey);
    }

    @Override
    public void send(EmailMessage message) {
        // Traduz do nosso modelo para o modelo do SendGrid
        Mail mail = new Mail(
            new Email(message.from()),
            message.subject(),
            new Email(message.to()),
            new Content("text/plain", message.body())
        );

        Request request = new Request();
        request.setMethod(Method.POST);
        request.setEndpoint("mail/send");

        try {
            request.setBody(mail.build());
            sendGridClient.api(request);
        } catch (IOException e) {
            throw new NotificationException("Falha ao enviar via SendGrid", e);
        }
    }
}
```

```java
// Adapter para AWS SES
@Component("aws-ses")
public class AwsSesEmailAdapter implements EmailProvider {

    private final SesClient sesClient;

    public AwsSesEmailAdapter(SesClient sesClient) {
        this.sesClient = sesClient;
    }

    @Override
    public void send(EmailMessage message) {
        // Traduz do nosso modelo para o modelo da AWS
        sesClient.sendEmail(SendEmailRequest.builder()
            .source(message.from())
            .destination(d -> d.toAddresses(message.to()))
            .message(m -> m
                .subject(c -> c.data(message.subject()))
                .body(b -> b.text(c -> c.data(message.body())))
            )
            .build()
        );
    }
}
```

```java
// Adapter para Twilio (SMS)
@Component("twilio")
public class TwilioSmsAdapter implements SmsProvider {

    public TwilioSmsAdapter(
        @Value("${twilio.account-sid}") String accountSid,
        @Value("${twilio.auth-token}")  String authToken
    ) {
        Twilio.init(accountSid, authToken);
    }

    @Override
    public void send(SmsMessage message) {
        // Traduz do nosso modelo para o modelo do Twilio
        Message.creator(
            new PhoneNumber(message.to()),
            new PhoneNumber(message.from()),
            message.body()
        ).create();
    }
}
```

> **Benefício :** Se o SendGrid mudar sua API,  altera apenas o `SendGridEmailAdapter`. O resto do sistema não é afetado.

## Factory: A Decisão em Tempo de Execução

### O que é

Factory centraliza a criação e seleção de objetos. Em vez de instanciar provedores espalhados pelo código, existe um único ponto responsável por decidir qual provedor usar.

### Por que usar aqui

Com múltiplos provedores, você precisa de uma lógica central para decidir qual usar seja por configuração, por tipo de mensagem, por disponibilidade ou até por fallback automático.

### Implementação em Java 21

```java
// Factory que seleciona o provedor pelo nome da configuração
@Component
@RequiredArgsConstructor
public class NotificationProviderFactory {

    // Spring injeta todos os beans que implementam EmailProvider
    private final Map<String, EmailProvider> emailProviders;
    private final Map<String, SmsProvider>   smsProviders;

    @Value("${notification.email.provider:sendgrid}")
    private String activeEmailProvider;

    @Value("${notification.sms.provider:twilio}")
    private String activeSmsProvider;

    public EmailProvider getEmailProvider() {
        return Optional.ofNullable(emailProviders.get(activeEmailProvider))
            .orElseThrow(() -> new IllegalStateException(
                "Provedor de e-mail não encontrado: " + activeEmailProvider
            ));
    }

    public SmsProvider getSmsProvider() {
        return Optional.ofNullable(smsProviders.get(activeSmsProvider))
            .orElseThrow(() -> new IllegalStateException(
                "Provedor de SMS não encontrado: " + activeSmsProvider
            ));
    }
}
```

```properties
# application.properties — troca de provedor sem alterar código
notification.email.provider=aws-ses
notification.sms.provider=twilio
```

> **Benefício :** Trocar de provedor em produção é uma mudança de configuração, não de código.

##  Padrão Chain of Responsibility para Fallback

### O que é e por que considerar

Em produção, provedores podem ficar indisponíveis. O **Chain of Responsibility** encadeia provedores em sequência se o principal falhar, o próximo da cadeia assume automaticamente.

```java
// Provedor com fallback automático usando Java 21
@Component
@RequiredArgsConstructor
public class FallbackEmailProvider implements EmailProvider {

    private final List<EmailProvider> providers; // SendGrid → AWS SES → Mailgun

    @Override
    public void send(EmailMessage message) {
        // Sealed classes do Java 21 para resultado tipado
        for (EmailProvider provider : providers) {
            try {
                provider.send(message);
                return; // sucesso — para aqui
            } catch (NotificationException e) {
                log.warn("Provedor {} falhou, tentando próximo...",
                    provider.getClass().getSimpleName(), e);
            }
        }
        throw new NotificationException("Todos os provedores falharam");
    }
}
```

## ✅ Por que essa combinação é a ideal?

| Princípio SOLID | Como é atendido |
|---|---|
| **S** — Single Responsibility | Cada Adapter cuida apenas de um provedor |
| **O** — Open/Closed | Adicionar novo provedor = nova classe, sem alterar as existentes |
| **L** — Liskov Substitution | Qualquer Adapter pode substituir outro sem quebrar o sistema |
| **I** — Interface Segregation | `EmailProvider` e `SmsProvider` são interfaces separadas e coesas |
| **D** — Dependency Inversion | `NotificationService` depende da abstração, não da implementação |
